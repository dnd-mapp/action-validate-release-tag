# Compile TypeScript scripts inside a composite action, not a bundled JS Action

This action's two checks (tag reachable from `main`, tag matches `package.json`'s version) are written in TypeScript, compiled with plain `tsc` (no bundler), and invoked from `action.yml` as ordinary composite steps (`run: node dist/verify-tag-reachable.js`), the same shape as `action-setup-workspace`'s existing `.mjs` scripts, just typed and unit-tested with Vitest instead of untested. `action.yml` is not switched to a bundled, single-entry-point JS Action (`runs: using: node24`) built on the `@actions/core` toolkit: this action takes no inputs and produces no outputs, so `@actions/core`'s main value, typed inputs/outputs, doesn't apply here, and staying composite keeps the two checks as separate, legible steps in the Actions UI rather than one opaque `run`. Since the compiled scripts only ever import Node's builtins, `tsc` alone is enough; a bundler like `@vercel/ncc` (`project-template`'s suggestion for a "JS/TS Action") exists purely to inline `node_modules`, and there's nothing here to inline.

`dist/` is committed to the repo: a composite action gets no install/build step when consumed via `uses:`, so committed output is the only way this works at all. It's kept in sync by both a `lint-staged` pre-commit rebuild and a CI step that rebuilds `dist/` fresh and diffs it against the committed copy, the same "local hook + CI backstop" pattern this org already uses for commitlint, since a skipped or bypassed hook is exactly the failure mode a lone backstop can't catch, and a lone hook can't catch a hook someone skipped.

`tsconfig.json` extends `@dnd-mapp/tsconfig/configs/node.json` rather than a handwritten local config, making this repo the first real consumer of that package from GitHub Package Registry. That requires `pull-request.yml` and `push-main.yml` to authenticate `pnpm i` against the `@dnd-mapp` scope, solved by forwarding `secrets.GITHUB_TOKEN` (with a `packages: read` permission) into `action-setup-workspace@v2.0.0`'s `node-auth-token` input, the same mechanism `action-setup-workspace` and `tsconfig` now use for themselves, so this repo isn't inventing a new pattern, just being the first consumer of an already-solved one.

Vitest, not Node's built-in `node --test`, runs the tests: this is the org's first Vitest usage, chosen for native TypeScript support (no separate transform step to wire up) and a richer assertion/mocking API than the built-in runner offers.

## Considered options

- **A bundled JS Action** (`@vercel/ncc`, `runs: using: node24`, `@actions/core`). Rejected: no inputs/outputs to justify the toolkit, and bundling would exist only to inline dependencies this action doesn't have.
- **A local, handwritten `tsconfig.json`**, avoiding the GitHub Package Registry authentication problem entirely. Rejected in favor of being the first real consumer of the shared preset, now that `action-setup-workspace@v2.0.0` and its accompanying CI wiring solve that authentication problem org-wide.
- **A CI-only `dist/` freshness check, no pre-commit rebuild.** Rejected: mirrors the commitlint precedent of pairing a local hook with a CI backstop, rather than relying on either alone.
