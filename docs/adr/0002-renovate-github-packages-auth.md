# Authenticate Renovate to GitHub Package Registry with a machine-account token, not the org's GitHub App

`.github/renovate.json` adds a `hostRules` entry authenticating `npm.pkg.github.com` with `{{ secrets.NODE_AUTH_TOKEN }}`, a classic PAT (`read:packages`) minted on a dedicated machine GitHub account created for `dnd-mapp` org automation, stored as an org-scoped secret in the Mend/Renovate hosted dashboard (`developer.mend.io`), not committed to this or any other repo in plaintext or encrypted form. The secret is named `NODE_AUTH_TOKEN` to match the existing `node-auth-token` input `action-setup-workspace` and this repo's own CI already use for the same `@dnd-mapp` scope, one name for the same kind of credential everywhere it shows up in this org, rather than inventing a Renovate-specific name. This repo is the first real consumer of `@dnd-mapp/tsconfig` as an installed `devDependency` ([ADR 0001](0001-typescript-compiled-composite-action.md)), and the hosted Renovate GitHub App could not resolve it: its "Package lookup failures" dashboard warning traced back to a GitHub platform limitation, not a config gap. GitHub App installation tokens are categorically rejected by `npm.pkg.github.com`, even when the app has `Packages: read` permission and is installed with access to the publishing repo, which the `dnd-mapp` org's own existing GitHub App already had. Only a classic PAT with `read:packages`, or the ephemeral `GITHUB_TOKEN` scoped to the package's own repo, are accepted; fine-grained PATs don't support Packages at all yet. A PAT was therefore unavoidable if Renovate stayed on the hosted App, and staying on the hosted App was itself a deliberate choice inherited from `dnd-mapp/tsconfig`'s ADR 0006: a self-hosted Actions workflow authenticating with `GITHUB_TOKEN` would work for reading packages, but its PRs wouldn't trigger `pull_request` checks like commitlint, which the org already decided it wanted.

The PAT belongs to a new machine account rather than a maintainer's personal account, so Renovate's access to `@dnd-mapp/*` packages doesn't depend on any one person's account standing or org membership. The secret is org-scoped in the Mend dashboard, not repo-scoped, so every current and future `dnd-mapp` repo that consumes an `@dnd-mapp/*` package can reference the same `hostRules` block without provisioning its own secret.

## Considered options

- **Rely on the org's existing GitHub App** (already has `Packages: read` and installation access to `dnd-mapp/tsconfig`). Rejected: not a configuration gap, GitHub App installation tokens cannot authenticate to `npm.pkg.github.com` at all, regardless of granted permissions.
- **Commit an encrypted PAT directly in `renovate.json`.** Rejected: still puts a long-lived credential's ciphertext in the repo and git history; the Mend dashboard secret keeps the raw token out of any repo entirely.
- **Use a maintainer's personal PAT.** Rejected: ties org-wide automation to one person's account; breaks if that account is rotated, restricted, or the person leaves.
- **Self-host Renovate via a GitHub Actions workflow using `GITHUB_TOKEN`.** Rejected, inherited from `dnd-mapp/tsconfig` ADR 0006: solves the package-auth problem but reintroduces the PR-trigger gap that decision already chose the hosted App to avoid.
- **Scope the Mend secret per-repo instead of org-wide.** Rejected: every `dnd-mapp` repo that later adds an `@dnd-mapp/*` dependency would repeat this same setup instead of just adding a `hostRules` block.
